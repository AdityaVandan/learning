Spent two years calling LLMs "hit or miss" before realizing the miss rate was mostly me.
As engineers we're used to being precise with machines. Function signatures, typed parameters, explicit contracts. Then we open a chat interface and suddenly start writing like we're texting a coworker. "Hey can you help me refactor this?" and then wonder why the output needs three follow-up corrections.
The model isn't vague. Your instruction is.
There are roughly three levels at which engineers actually use these things, and the output quality difference between them is not small.
Level one is conversational. You describe what you want in plain prose. Works for throwaway tasks. Breaks down the moment the task has multiple constraints, a specific format requirement, or depends on context the model has to infer rather than read. You're leaving too many decisions to the model and then blaming the model for making them.
Level two is structured prompting. This is where your instincts as an engineer start to pay off. XML is particularly effective here — wrapping context in named tags (<role>, <task>, <constraints>, <examples>) gives the model a clear schema to work against. It's not magic. It's the same reason typed function signatures produce fewer bugs than positional args with no names. JSON works well when you're chaining calls or need deterministic output shapes. Markdown headers impose hierarchy on long prompts the same way sectioning a design doc makes it readable. The structure isn't decoration. It changes how the model weights and processes the content.
Level three is treating the prompt like an interface contract. You define the role up front. You separate examples from instructions. You specify what "correct" looks like, not just what the task is. You control output format so whatever's consuming the response doesn't need to guess. And you iterate the way you'd iterate on any system component — isolate one variable, observe the change, adjust.
The engineers I've seen get the most out of these tools are the ones who applied the same rigor they'd bring to writing a good API. Clear inputs. Explicit expectations. No ambiguous side effects.
If you want one concrete thing to try: stop writing your prompt as a single paragraph. Break it into sections. Role, context, task, constraints. Even in plain English, that structure alone cuts down on garbage output more than any model switch will.
The interface looks simple. That's the trap.
